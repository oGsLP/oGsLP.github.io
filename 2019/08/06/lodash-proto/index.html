从Lodash的安全漏洞看JS的原型污染

  Lodash开源库爆出严重漏洞问题已经过去快一个月了，当时可以说瞬间燃爆Github的JS项目，Lodash维护者连夜修复Bug，Github也给每个波及用户high vulnerability提醒，强烈要求升级该包。虽然自己没主动用过Lodash，但自己的package.json里面依赖的大小型项目都用到了它，自己也不得不更新了几个好久没碰过的项目的依赖。那么这次安全漏洞到底是什么，来一起看看。

PS：这里不谈Lodash用法，需要请移步 Lodash官网

问题API

  • 最容易想到的漏洞和攻击就是注入式的，Lodash这次的漏洞也不例外，如果使用以下代码

    了解熟悉js原型的自然知道这样就形成了对Object的原型污染，当我们调用Object.prototype.toString就无法正常使用了

原型污染

  • 首先需要了解JS原型和原型链的知识，需要请移步 TO ADD，然后看一个例子

  每个对象都有一个 toString() 方法，当该对象被表示为一个文本值时，或者一个对象以预期的字符串方式引用时自动调用。默认情况下，toString() 方法被每个 Object 对象继承。如果此方法在自定义对象中未被覆盖，toString() 返回 [object type]，其中 type 是对象的类型。如果 Object 原型上的 toString 被污染，后果可想而知

Lodash 的修复方法

  显而易见，Lodash的维护人员的方法简单而有效，当遍历merge时如果出现’constructor’、’__proto__‘等敏感属性时则直接return，不给污染可乘之机

  那么面对可能的原型污染一般有什么样的方法可以解决呢？

原型污染的修复方法

  1. Object.freeze
    • bject.freeze() 方法可以冻结一个对象。一个被冻结的对象再也不能被修改；冻结了一个对象则不能向这个对象添加新的属性，不能删除已有属性，不能修改该对象已有属性的可枚举性、可配置性、可写性，以及不能修改已有属性的值。此外，冻结一个对象后该对象的原型也不能被修改。freeze() 返回和传入的参数相同的对象
  2. 建立JSON schema在解析用户输入内容是，通过JSON schema过滤敏感键名。规避不安全的递归性合并这一点类似lodash修复手段，完善了合并操作的安全性，对敏感键名跳过处理
  3. 使用无原型对象，在创建对象时，不采用字面量方式，而是使用Object.create(null)：
    • Object.create()方法创建一个新对象，使用现有的对象来提供新创建的对象的proto
    • Object.create(null)的返回值不会链接到Object.prototype
  4. Map代替Object
    • Map 对象保存键/值对，是键/值对的集合。任何值（对象或者原始值）都可以作为一个键或一个值。使用Map数据结构，不会存在Object原型污染状况
    • Object 的键只支持String或者Symbols两种类型，Map的键可以是任意值，包括函数、对象、基本类型
    • Map中的键值是有序的，而Object中的键则不是
    • 具体API上的差异：比如，通过size属性直接获取一个Map的键值对个数，而Object的键值无法获取；再比如迭代一个Map和Object差异也比较明显
    • Map在频繁增删键值对的场景下会有些性能优势

总结

  用不同的语言写出功能相同的程序，看上去都差不多而且没有问题，但会因为各种语言特有的性质从而暴露出不同问题，就像JS特有的原型prototype，深入了解一门语言的原理并在编程时时刻注意，让自己的程序不仅是能用还要可靠。
